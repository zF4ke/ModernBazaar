# Modern Bazaar — Codebase Improvement Audit

## Executive Summary

This audit consolidates findings across six dimensions of the Modern Bazaar codebase: backend code quality, financial-math correctness, performance/database efficiency, frontend UI/UX, frontend data/type safety, and infrastructure/testing/CI. The most urgent risks are **financial correctness** (the flipping profit model omits the in-game bazaar tax and applies risk penalties twice, overstating profitability) and **test coverage** (the core scoring algorithm — the product's central value proposition — has zero tests). Several high-value, low-effort hygiene wins are immediately available: deleting a stray scratch file, removing debug logging, and hardening cache keys. The recommendations below are deduped, grouped by area, and ordered by severity, with a suggested sequencing at the end.

## Backend — Financial Math & Metrics Correctness

- **Missing Bazaar Tax in Profit Calculations** (high/moderate) — `FlippingScorer.java:226` (and `FinanceMetricsService`) — `profitPerItem = spread * (1 - riskScore)` treats the full spread as realizable profit, ignoring the ~1.125%/side bazaar tax; the error compounds through `reasonableProfitPerHour` and `profitPerHour`. — Deduct fees explicitly, e.g. `profitPerItem = max(0, spread * (1 - taxBuy - taxSell) * (1 - riskScore))`; define `BAZAAR_BUY_FEE`/`BAZAAR_SELL_FEE` in `ScoringConstants`, verify the rate against gameplay, and document the assumption. *(Three auditors flagged this with slightly different rates — 1% vs 1.125%/side; consolidated here as one issue. Confirm the correct in-game rate.)*

- **Scoring Monotonicity Claim Partially False (Risk applied twice)** (high/moderate) — `FlippingScorer.java:30-34` (javadoc) vs `226` & `271` — Risk reduces profit at line 226 (`spread * (1 - riskScore)`) *and* again as a final penalty at line 271 (`riskScore * 0.1`). A `riskScore=0.8` item yields ~0.184× the base value, not the documented "max 10% reduction"; combined with `liquidityFactor` the score is non-monotonic in risk. — Decide on a single risk-application path: either reduce profit margins only, or apply a light score penalty only. Remove the redundant term and correct the javadoc. *(Duplicate finding "Risk Score Penalizes Profit Twice" at 226/271 is the same issue — merged.)*

- **Budget Per-Hour Calculation Unit Confusion** (high/quick) — `FlippingScorer.java:214-220` — `maxAffordablePerHour = floor(budget / (is * horizon))` produces units-per-hour but the naming/formula reads as total-over-horizon; it is then min'd against `throughputPerHour`, mixing time bases. Result is correct but deceptive. — Add an explicit comment documenting the time-basis (`budget / (price × horizonHours) = units/hour`); consider renaming.

- **Risk Threshold Asymmetry / Unexplained 0.85 Floor** (medium/moderate) — `RiskToolkit.java:25-31` — `manipulatedLikely` triggers on `baseDevMax ≥ 0.15` or `shockMax ≥ 0.12`, while risk maps via `/0.20` and `/0.25`; the `0.85` floor is undocumented and possibly redundant (15% dev already maps to >0.75). — Clarify and unify the trigger conditions; document or remove the `0.85` floor; add unit tests with concrete examples.

- **Competition Penalty Amplification / Coefficient Drift** (medium/moderate) — `FlippingScorer.java:274` — `competitionPenalty = min(0.10, effectiveChurn * 0.0007)`; the coefficient appears to have drifted from a documented `0.0005`, and the `0.5` imbalance multiplier's relation to market risk is unvalidated. — Document the formula and justify `0.0007`/`0.5` with examples; add unit tests; note if empirical.

- **Division-by-Zero Masked by `horizonHours` Default** (medium/quick) — `FlippingScorer.java:210` — Null/zero/negative `horizonHours` silently clamps to `1.0` with no signal, hiding caller bugs. — Reject/warn on missing horizon, or document the clamp contract in the `Inputs` javadoc.

- **Demand/Supply `minPos` Selection Undefined** (medium/moderate) — `FlippingScorer.java:367-369` — `demand`/`supply` use the minimum positive across 3 windows (most pessimistic) without rationale; `flowProxy = demand + supply` differs from actual throughput `min(demand, supply)` and is only passed to `FinanceAverages`. — Document why `minPos` (conservative) is used; clarify whether `flowProxy` is diagnostic; add unit tests against game mechanics.

- **NaN/Infinity Risk in Effective Churn** (medium/quick) — `FlippingScorer.java:189-191` — No `isFinite` guard on `churnBuy`/`churnSell`; corrupt data could propagate Infinity through `effectiveChurn`. — Clamp non-finite churn inputs to `0.0` (here or upstream in the aggregation service).

- **Unweighted Mean on Sparse Windowed Data** (low/moderate) — `FinanceMetricsService.java:127-164` and `BazaarFinanceMetricsAggregationService.java:72-98` — Simple arithmetic means ignore time/volume weighting; sparse or outlier hours skew results. — Document the unweighted/sparse behavior; optionally add volume weighting or outlier capping and flag items with <80% window coverage.

- **Partial/Truncated Window Data Not Flagged** (low/quick) — `FinanceMetricsService.java:94-95` — When fewer rows than the requested window exist, the average is computed over a partial window but treated downstream as full. — Add an `actualHours`/`fullyPopulated` flag to `FinanceAverages` and discount or skip incomplete windows. *(Same root cause as the null-safety finding above.)*

- **Hard-Coded Liquidity Threshold `8.0`** (low/quick) — `FlippingScorer.java:277` — `liquidityFactor = min(1.0, throughputPerHour / 8.0)`; the `8.0` "good" threshold is unexplained and coupled to the `≥1` gating at line 204. — Document/parameterize the threshold and assert the throughput invariant. *(Folds into `ScoringConstants`.)*

- **Suggested Units Can Round to Zero** (low/quick) — `FlippingScorer.java:236-240` — High competition can drive `suggestedUnitsPerHour` to ~0, surfacing an opportunity with zero tradable units. — If below a minimum-viable unit, return `Score(0)` or log; document Score-field contract when units are zero.

## Backend — Performance & Database Efficiency

- **Unbounded Cache Key Growth in `flipOpportunities`** (high/quick) — `StrategyFlippingService.java:21,62` — Keys concatenate filter `toString()` (a record with nullable fields and a free-text `q`), producing very long, unbounded keys. — Use an explicit, stable key built from individual filter fields plus sort/budget/horizon and the advanced params; consider `@CacheEvict` on scheduler jobs.

- **Cache Key Collision from `filter.hashCode()`** (high/quick) — `BazaarItemsQueryService.java:37` — `hashCode()`-based key risks collisions with nullable fields, and including `#page` fragments the cache per page. — Build an explicit field-based key and drop `#page` (cache full result sets, slice in memory).

- **Cache TTL Inconsistency & Heap Pressure** (medium/quick) — `CacheConfig.java:31-81` — All caches use `maximumSize=10_000` × 5-8 caches → 50-80k entries; disabled TTL `@Value`s leave confusing magic `300s` defaults. — Size caches by distinct product count, wire TTL `@Value`s, tune per-cache TTLs (list ~60-120s, item detail ~5min), enable Caffeine `recordStats()`.

- **N+1 on Lazy `SkyblockItem` in `preloadNames`** (medium/moderate) — `BazaarItemsQueryService.java:344-350,475-481` (and FlippingScorer) — `getSkyblockItem().getName()` on each `BazaarItem` triggers a SELECT per item (50 items → 50 queries). — Add `@EntityGraph(attributePaths={"skyblockItem"})` to `findAllByProductIdIn`, or use a native `(productId, name)` projection.

- **Parallel Stream Without Thread-Safety Guarantees** (medium/moderate) — `FlippingScorer.java:348-428` — `parallelStream()` (≥128 items) calls `FinanceMetricsService` whose caching/DB access may not be thread-safe; the main `getMultiWindowAverages()` fetch (line 334) is sequential anyway. — Either drop the parallel stream and rely on HikariCP for I/O parallelism, or verify thread-safety, move the fetch inside the stream, and bound the ForkJoinPool.

- **Missing Indexes for Cascade Deletes** (medium/moderate) — `db/migration/V4__more_perf_indexes.sql:8-9` (and `BazaarProductSnapshotRepository:189-216`) — Retention deletes scan `bazaar_order_entry` and the snapshot subquery without optimal indexes. — Add `idx_snapshot_fetched_at_id (fetched_at, id)` and an explicit `(snapshot_id)` index on `bazaar_order_entry`.

- **Possibly Missing FK Index on `bazaar_hour_point`** (low/moderate) — `V1__baseline_schema.sql:76-95`, `BazaarHourPointRepository.java:37` — Join to `BazaarItemHourSummary` for the cutoff filter may lack a FK index. — Verify/add `idx_hour_point_summary_id` and a `(hour_start)` index on the summary table. *(Related to the cascade-delete index gap above.)*

- **Long-Held Locks in Batch Deletions** (medium/moderate) — `BazaarSnapshotsRetentionJob.java:50-69` — Three large sequential `@Modifying` deletes in one transaction can hold locks for seconds, blocking reads. — Delete in bounded batches (`LIMIT 10000` in a loop with short pauses) to reduce contention.

- **Three Separate COUNT Queries Before Deletion** (medium/quick) — `BazaarSnapshotsRetentionJob.java:58-61` — Informational counts add ~100-300ms latency before deletion. — Guard with `log.isInfoEnabled()` or log row counts returned by the DELETE statements instead.

- **`findRangeWithPoints` `@EntityGraph` + `join fetch` Conflict** (medium/moderate) — `BazaarItemHourSummaryRepository.java:33-46` — Combining `@EntityGraph(points)` with `left join fetch hs.points` can double-fetch or duplicate rows. — Keep only the `join fetch` with `distinct`, or use a DTO projection.

- **Inefficient Per-Metric Stream Averaging** (low/quick) — `BazaarFinanceMetricsAggregationService.java:72-99` — 16 metrics × ~50 items → ~800 stream passes per hourly run, plus an inline `ToDouble` interface. — Accumulate all metrics in a single pass over the list, dividing once.

- **`AdminPlansController.list()` Unpaginated `findAll()`** (low/quick) — `AdminPlansController.java:40-44` — Loads all plans into memory with no pagination. — Add `Pageable` pagination, or at minimum a size-threshold warning.

## Recommended Sequencing

Start with the **Quick Wins** — delete the scratch file, strip debug logging, swap `System.out` for slf4j, and remove the dead method and `as any` casts — since they are near-zero-risk and shrink the surface area for everything else. Next, treat **financial correctness as a paired effort with testing**: stand up the `FlippingScorer` test suite *first*, then land the bazaar-tax, double-risk-penalty, and unit-confusion fixes behind those tests so the changes are pinned by regressions; bundle the `ScoringConstants` extraction (magic numbers, window hours, liquidity/competition thresholds) into the same pass. In parallel, knock out the **cache-key and N+1 performance fixes**, which are low-effort but protect production stability, followed by the retention-job index/batching work. Then take on the **frontend type-safety initiative** as one coordinated change (shared `ErrorResponse` type + `isBackendError` guard + `AppError`), which dissolves most of the scattered `any` findings at once, and pair it with component extraction and the new frontend test harness. Finish with the **infra/observability** items (CI coverage reporting, integration/smoke tests, custom metrics, Prometheus auth), which harden the pipeline once the code-level debt is paid down.
